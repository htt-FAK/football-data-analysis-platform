## 1. 项目概述

### 1.1 项目背景

体育赛事数据在战术复盘、媒体传播和竞技训练中的价值日益凸显。当前赛事数据分散在多个平台，统计口径不统一，非结构化信息难以直接用于分析。本项目旨在构建一套完整的体育赛事数据采集与分析平台，实现从数据采集、清洗存储到分析可视化的大数据全链路处理。

### 1.2 项目目标

> 聚焦海量体育赛事数据的自动化采集与深度解析，将非结构化赛事信息转化为标准化数据资产，通过多维度挖掘（攻防特征、联赛格局、球员价值等），为战术复盘、媒体传播和竞技训练提供科学、直观的数据支撑，让比赛细节"有据可依"。

**具体目标：**

1. 实现多源自动化数据采集，覆盖 2026 年世界杯及英超联赛的赛程、赛事、积分榜、球员技术统计等
2. 构建数据清洗与标准化管道，统一多平台数据口径，保障数据质量
3. 建立多维度分析模型，深度挖掘联赛竞争格局、球队攻防效率及球员综合能力
4. 输出可视化报告，自动生成攻防雷达图、积分榜趋势、球员能力矩阵等图表
5. **支持近实时数据更新：比赛进行中每 30 秒自动刷新比分和事件，用户前端可感知最新战况**
6. **扩展数据源至 8+ 个，包含 API-Football、TheSportsDB、OpenLigaDB 等稳定数据源**
7. **增加数据源监控面板，实时展示各数据源健康状态和采集日志**

| **爬虫框架** | requests + BeautifulSoup | - | 网页数据采集 |
| **调度框架** | APScheduler | 3.10+ | 定时爬虫任务调度 + 30 秒实时轮询 |
| **数据分析** | Pandas + Scikit-learn | - | 数据清洗与模型训练 |
| **HDFS 客户端** | hdfs (Python) | - | Python 操作 HDFS |
| **前端框架** | React + TypeScript | 18+ | 单页应用（含实时刷新页面） |
| **前端构建** | Vite | 5+ | 开发与构建 |
| **UI 组件库** | Ant Design | 5+ | 页面组件 |
| **可视化图表** | ECharts | 5+ | 数据可视化 |
| **HTTP 客户端** | Axios | - | 前端 API 调用 |

### 2.3 技术选型理由

- **Hadoop HDFS**：课设要求大数据存储组件，HDFS 伪分布式满足要求，Parquet 列式存储适合分析场景
- **FastAPI**：异步高性能，自带 Swagger 文档，适合数据 API 开发
- **React + TypeScript**：类型安全，生态丰富，ECharts 可视化能力强
- **MySQL**：关系型存储业务查询数据，配合 HDFS 形成分层存储
- **APScheduler**：轻量级 Python 调度，无需引入 Airflow 等重量级组件

| 项目 | 现状 | 是否需要调整 |
|------|------|-------------|
| 操作系统 | TencentOS Server 3 (CentOS 系) | 不需要 |
| Java | OpenJDK 1.8.0 | **不需要升级**（Hadoop 3.3.6 官方推荐 JDK 8） |
| Hadoop | 3.3.6 伪分布式（已部署） | 不需要，直接复用 |
| MySQL | 已有：118.126.102.143 / root1 | 不需要安装，直接复用 |
| Python | 待确认 | 需安装 3.8+ 及虚拟环境 |
| Node.js | 待安装 | 需安装 18+ |

> **重要警告**：不要升级到 JDK 11。Hadoop 3.3.6 官方推荐 JDK 8，升级会导致兼容性问题，需要重新格式化 NameNode 并重建所有数据。

> **Hive 决策**：不引入 Hive。HDFS 存储 + Python Pandas 分析足够覆盖课设要求。Hive 会增加 MySQL 元数据库配置复杂度、schematool 初始化失败风险和学习成本。
>
> **MySQL 决策**：复用已有 MySQL 实例（118.126.102.143），在 root1 数据库中新增体育相关表，不删除现有表。

### 3.8 增量更新机制

为避免每次全量覆盖、降低对数据源的压力并支持版本化快照，系统采用基于 `data_hash` 的增量 upsert 机制：

```
                  ┌─────────────────────────────────────┐
   爬虫/API 拉取  │  1. 计算每行字段拼接的 data_hash      │
        ────────▶ │  2. INSERT ... ON DUPLICATE KEY UPDATE│
                  │     - 命中且 hash 不变 → 跳过（版本不变）│
                  │     - 命中且 hash 变化 → 更新字段，     │
                  │       version = version + 1          │
                  │  3. 写 crawl_logs 记录本次抓取元信息   │
                  │  4. 仅对有变更的记录发布 Redis 通知    │
                  └─────────────────────────────────────┘
                                    │
                                    ▼
                  HDFS 按版本号快照（仅变化批次写入）
```

**核心要点**：

| 环节 | 实现 | 说明 |
|------|------|------|
| 去重判定 | `data_hash`（字段拼接后 SHA256 取前 16 位） | 内容未变则不更新，节省写开销 |
| 幂等写入 | `INSERT ... ON DUPLICATE KEY UPDATE ... version = version + 1` | MySQL 唯一键保证并发安全 |
| 数据源溯源 | 每表新增 `data_source`、`source_id`、`last_updated_at`、`version` 字段 | 可追溯每条记录来自哪个源、何时更新 |
| 抓取留痕 | `crawl_logs` 表记录每次任务的源、数量、耗时、成功/失败 | 供数据源监控页（第 7 章）展示 |
| 变更通知 | 变更记录通过 Redis `PUBLISH data:update <payload>` | 触发 WebSocket 推送给前端 |
| HDFS 快照 | 仅当某表有变更时，按 `table_name/version/date` 追加一份 Parquet | 既保留历史版本，又避免无效 I/O |

### 3.9 实时推送通道（WebSocket + Redis）

前端不再依赖整页轮询，而是通过 WebSocket 接收增量推送，并结合 Redis 缓存实现秒级实时比分。

```
  APScheduler (每 30s)              Redis                     FastAPI WebSocket
  ┌───────────────┐      upsert    ┌───────────────┐  SUBSCRIBE  ┌──────────────────┐
  │ 爬虫拉取实时   │──────────────▶│  live:{match} │◀────────────│  ConnectionManager │
  │ 比赛比分/事件  │  PUBLISH      │  (比分缓存)    │             │  按联赛分组的连接池 │
  │               │  data:update  │               │             │  心跳 25s / 断线重连│
  └───────────────┘     ──────▶───┴───────────────┘   推送变更   └────────┬─────────┘
                                                                           │
                                                                           ▼
                                                                  浏览器 useLiveMatch
                                                                  （WebSocket 优先，
                                                                   失败降级为 30s 轮询）
```

**设计要点**：

- **订阅粒度**：客户端连接 `/ws` 后发送 `{"action":"subscribe","league_ids":[...]}`，仅接收订阅联赛内的比赛/事件/积分变动。
- **降级策略**：WebSocket 断开或 25s 心跳无响应时，前端 `useWebSocket` Hook 自动回退到 `GET /api/matches/live` 30 秒轮询，保证可用性。
- **缓存层**：进行中比赛的比分、最后事件、控球率等高频字段先写 Redis（TTL 5 分钟），前端首屏直接读缓存，减轻 MySQL 压力。
- **并发控制**：FastAPI `ConnectionManager` 用 `asyncio` 管理连接池，Redis Pub/Sub 单消费者广播，避免逐连接查库。

- 表格结构规整，解析难度低
- 数据质量高，包含 xG 等高级指标

#### 4.2.5 API-Football（实时比分主源，API）

```
采集内容
├── 实时数据
│   ├── 进行中比赛比分 / 状态（未开始/进行中/已结束）
│   ├── 进球 / 换人 / 红黄牌事件（带分钟）
│   └── 比赛阵容 / 阵型

│   └── 按联赛 + 日期 / 轮次查询
```

- 调用频率受免费配额限制，仅在"有比赛进行中"时启用 30 秒高频轮询，其余时段降频

### 4.3 采集调度策略

调度策略分四档，由 APScheduler 的 `CronTrigger` / `IntervalTrigger` 驱动，并接通实时推送通道（3.9）：

```
调度计划（v2）
═══════════════════════════════════════════════════════

实时高频（每 30 秒，仅当有比赛进行中）:
├── API-Football  — /fixtures?live=all 拉取实时比分/事件
├── 懂球帝        — 实时比分兜底（API-Football 配额耗尽时启用）
└── → upsert MySQL + 写 Redis 缓存 + PUBLISH data:update
    → WebSocket 推送给订阅了对应联赛的前端连接

每日采集（每天 06:00）:
├── 懂球帝        — 赛程更新、积分榜刷新
├── FBref         — 球队/球员统计更新
├── Understat     — 射门数据更新
├── API-Football  — 当日赛程 / 阵容预取
└── TheSportsDB   — 徽标 / 元信息刷新

每周采集（每周一 03:00）:
├── FBref         — 完整周度统计刷新
├── Football-Data — 下载最新赛季 CSV
├── OpenLigaDB    — 周度事件回填
└── TeamRankings  — 球队评分 / 概率刷新

按需采集（手动触发 / API 调用）:
├── 历史赛季数据批量导入
└── 单场比赛详情即时拉取（前端点开比赛卡片时触发）
```

**关键改进**：相比 v1，新增"实时高频"档，把进行中比赛从 5 分钟一次提升到 30 秒一次，并通过 Redis Pub/Sub → WebSocket 链路秒级触达前端；非比赛时段自动降频，兼顾实时性与数据源配额。

### 4.4 反爬策略

```
反爬应对措施
├── 请求频率控制
│   └── 每次请求间隔 2-5 秒，随机抖动；API 类源遵守官方配额
├── User-Agent 轮换
│   └── 维护 UA 池，每次随机选取
├── Referer 头设置
│   └── 模拟正常浏览行为
├── 重试机制
│   └── 最多重试 3 次，指数退避
├── 代理 IP（可选）
│   └── 如遇 IP 封禁，使用代理池
├── API Key 管理
│   └── API-Football / TheSportsDB 的 key 存 .env，按月轮换
└── 数据缓存
    └── 已采集数据缓存 + data_hash 去重，避免重复请求与重复写入
```

### 4.5 数据融合策略（多源融合与降级）

同一实体（球队/比赛/球员）可能来自多个源，系统按"取信优先级 + 字段级补全"融合：

```
融合规则
├── 赛果 / 比分       → API-Football 优先，懂球帝 / Football-Data 校验
├── 高级统计 / xG     → FBref / Understat 优先（数据更专业）
├── 徽标 / 元信息      → TheSportsDB 优先，懂球帝补图
├── 事件（进球/换人）   → API-Football 优先，OpenLigaDB 兜底
└── 评分 / 概率       → TeamRankings（独立维度，单独展示）

降级链（某源失败时）
├── API-Football 实时失败 → 降级到懂球帝实时比分 → 再降级为 30s 轮询只读 Redis 缓存
├── FBref 失败          → 用 API-Football 统计端点兜底
└── 任意源失败           → 记入 crawl_logs（status=failed），监控页标红，不影响其他源
```

每条记录的 `data_source` 字段标注其真实来源，便于在监控页追溯与比对；不同源对同一实体的冲突字段以"高优先级源覆盖"为准，保留 `version` 与 `last_updated_at` 以支撑审计。

### 6.4 增量变更检测

增量变更检测是 3.8 增量更新机制的上游，决定"哪些行需要进入 upsert 流程"。核心目标是：**只把真正变化的记录送进数据库，避免全量覆盖和无效 I/O**。

```
检测流程
═══════════════════════════════════════════════════════

爬虫/API 拉取新批次
        │
        ▼
按唯一键（如 match_id + source）分组
        │
        ▼
逐行计算 data_hash = SHA256(字段拼接).substring(0, 16)
        │
        ▼
与 MySQL 现有行的 data_hash 比对
        │
        ├── 命中且 hash 相同  →  跳过（无变化，不写库）
        ├── 命中且 hash 不同  →  标记为 update（进入 upsert，version + 1）
        └── 未命中            →  标记为 insert（进入 upsert，version = 1）
        │
        ▼
仅把 change_set（insert + update）写入数据库
        │
        ▼
对 update 的记录发布 Redis PUBLISH data:update
        │
        ▼
WebSocket 推送给订阅了对应联赛的前端连接
```

**实现要点**：

| 环节 | 实现 | 说明 |
|------|------|------|
| 唯一键 | `uk_<table>_source`（如 `uk_match_source`） | 同一实体同一来源只有一行，保证幂等 |
| 指纹算法 | `SHA256(concatenated_fields)[:16]` | 16 位 hex 足够防碰撞，比全量 SHA256 省空间 |
| 比对时机 | 写入 MySQL 之前，在 Python 内存中完成 | 不依赖 MySQL 触发器，逻辑集中 |
| 变更粒度 | 行级（record-level） | 不追踪具体哪个字段变了，只判断整行是否变化 |
| 通知触发 | 仅 `update` 类型的记录触发 Redis Pub/Sub | `insert` 不触发（新比赛已在赛程页展示，无需全局通知） |
| HDFS 快照 | 仅当 `change_set` 非空时，按 `table/version/date` 追加 Parquet | 无变化批次不写 HDFS，节省存储和计算 |

**边界处理**：
- **删除场景**：数据源未返回某场比赛（如已取消），系统不做物理删除，仅标记 `status='cancelled'` 并记录 `last_updated_at`，保留历史可追溯。
- **字段缺失**：源端返回字段不完整时，缺失字段保留旧值（若已存在）或为 `NULL`（若为新记录），不阻断整行写入。
- **并发安全**：`INSERT ... ON DUPLICATE KEY UPDATE` 在 MySQL InnoDB 下是原子操作，多爬虫任务同时 upsert 同一行不会出现脏写。

## 7. 多维度分析模型设计

### 7.1 联赛竞争格局分析

**目标**：量化联赛中各球队的竞争力分布和排名稳定性。

```
分析维度
├── 积分分布
│   ├── 积分差距分析（榜首与尾队、前四与中游）
│   └── 积分集中度（标准差、基尼系数）
├── 排名波动
│   ├── 各球队排名变化标准差
│   └── 排名上升/下降趋势
├── 分组竞争
│   ├── 欧战区（前 4-6 名）竞争激烈度
│   ├── 中游集团分析
│   └── 保级区（后 3 名）竞争
└── 主客场差异
    ├── 主场胜率 vs 客场胜率
    └── 主场积分贡献占比
```

**输出**：联赛竞争格局热力图、排名波动折线图、分组竞争雷达图。

### 7.2 球队攻防效率分析

**目标**：从进攻和防守两个维度评估球队综合实力。

```
进攻效率指标
├── 进球数 / 射门数 = 射门转化率
├── xG vs 实际进球 = 进球效率（高于 xG 说明把握机会能力强）
├── 场均射门 / 射正
└── 关键传球 / 创造机会数

防守稳固度指标
├── 失球数 / 被射门数 = 防守转化率
├── xGA vs 实际失球 = 防守效率
├── 场均被射门 / 被射正
├── 抢断 / 拦截 / 解围
└── 零封场次比例

综合评级
├── 进攻评分（0-100）：加权进攻指标
├── 防守评分（0-100）：加权防守指标
└── 综合评分 = 0.5 × 进攻评分 + 0.5 × 防守评分
```

**输出**：攻防雷达图、xG vs 实际进球散点图、攻防效率排行榜。

### 7.3 球员综合能力评分

**目标**：基于多维度数据对球员进行综合评分，并按位置（GK/DF/MF/FW）展示差异化六边图。

```
评分维度（统一基础维度，按位置差异化展示）
├── 进攻能力
│   ├── 进球、助攻、xG、xA
│   ├── 射门转化率、关键传球
│   └── 进攻三区触球
├── 组织能力
│   ├── 传球数、传球成功率
│   ├── 关键传球、创造机会
│   └── 控球、过人
├── 防守能力
│   ├── 抢断、拦截、解围
│   ├── 防守对抗成功率
│   └── 空中对抗
├── 门线能力（仅 GK 展示）
│   ├── 扑救数、扑救率
│   ├── 预期扑救（xCS）
│   └── 出击次数、手抛球/开球成功率
├── 身体/运动能力
│   ├── 冲刺速度、跑动距离
│   ├── 对抗成功率
│   └── 弹跳（门将）/ 耐力
└── 纪律与出场
    ├── 黄红牌
    ├── 出场时间、出勤率
    └── 稳定性（评分方差）

六边图维度配置（按位置）
├── GK（门将）：扑救反应 + 出球能力 + 防守站位 + 组织策应 + 纪律 + 稳定性
├── DF（后卫）：防守抢断 + 空中对抗 + 出球组织 + 跑动覆盖 + 进攻插上 + 纪律
├── MF（中场）：传球组织 + 防守拦截 + 进攻创造力 + 跑动覆盖 + 关键传球 + 纪律
└── FW（前锋）：射门终结 + 跑位穿插 + 关键传球 + 身体对抗 + 进攻效率 + 纪律

位置权重（综合评分用）
├── GK（门将）：防守 70% + 组织 20% + 纪律 10%
├── DF（后卫）：防守 50% + 组织 25% + 进攻 15% + 纪律 10%
├── MF（中场）：组织 40% + 进攻 30% + 防守 20% + 纪律 10%
└── FW（前锋）：进攻 60% + 组织 25% + 纪律 15%
```

**输出**：
- 球员能力雷达图（按位置展示不同六边图维度）
- 球员能力矩阵散点图（按位置分组着色）
- 位置排行榜（同位置球员横向对比）
- 球员对比工具（跨位置或同位置对比，雷达图叠加展示）

### 7.4 xG 预期进球模型

**目标**：基于射门特征预测进球概率，评估球队和球员的射门质量。

```
数据来源
└── Understat 提供现成 xG 值（每脚射门附带 xG 标签）
    ├── 直接用于分析展示
    └── 可选：用逻辑回归做建模演示

分析维度
├── 球队层面
│   ├── 场均 xG vs 实际进球（进攻效率）
│   ├── 场均 xGA vs 实际失球（防守效率）
│   └── xG 累计曲线（比赛走势）
├── 球员层面
│   ├── 球员 xG vs 实际进球（终结能力）
│   ├── 场均 xG（机会创造）
│   └── xG 排行榜
└── 比赛层面
    ├── 双方 xG 时间线对比
    ├── 射门位置 xG 热图
    └── 关键射门 xG 标注

可选建模演示（逻辑回归）
├── 特征：射门距离、角度、类型、情境
├── 标签：进球 / 未进球
├── 评估：ROC-AUC, Brier Score
└── 用途：展示数据建模过程，不影响主流程
```

### 7.5 比赛关键事件影响量化

**目标**：量化比赛中关键事件（进球、红牌、换人等）对比赛走势的影响。

```
事件影响度量
├── 进球影响
│   ├── 进球前后 xG 速率变化
│   ├── 进球后控球率变化
│   └── 领先后防守强度变化
├── 红牌影响
│   ├── 少一人后 xG 下降幅度
│   └── 十人应阵的防守策略变化
├── 换人影响
│   ├── 换人前后球队表现差异
│   └── 替补球员贡献度
└── 比赛 momentum
    ├── 基于 xG 时间线的势头分析
    └── 比赛转折点识别
```

**输出**：比赛事件时间线图、xG 累计曲线、momentum 热力带。

**核心交互**：
- **双人对比模式**：顶部两个下拉框分别选择球员 A 和球员 B，系统自动识别位置并切换对应六边图维度；若位置不同，展示**公共维度雷达图**并标注差异提示。

- **赛季数据对比**：并排展示进球、助攻、出场时间、xG、评分等核心赛季数据。

#### WebSocket 协议

客户端连接 `ws://<host>:8000/ws` 后，通过 JSON 消息进行交互：

```
客户端 → 服务端
{
  "action": "subscribe",        // 订阅联赛比赛更新
  "league_ids": [39, 140, 135] // 英超、西甲、意甲
}

{
  "action": "unsubscribe",      // 取消订阅
  "league_ids": [39]
}

{
  "action": "ping"              // 心跳（服务端 25s 无推送时客户端主动 ping）
}

服务端 → 客户端（推送消息）
{
  "type": "match_update",       // 比赛数据更新
  "data": { "match_id": 123, "score": "2-1", "minute": 67, ... }
}

{
  "type": "event",              // 比赛事件（进球/红牌/换人）
  "data": { "match_id": 123, "event_type": "goal", "player": "...", "minute": 67 }
}

{
  "type": "standing_update",    // 积分榜变动
  "data": { "league_id": 39, "standings": [...] }
}

{
  "type": "pong"                // 心跳响应
}
```

**连接管理**：
- 服务端 `ConnectionManager` 按 `league_id` 分组管理连接，仅向订阅了对应联赛的连接推送相关消息，减少无效传输。
- 客户端无任何消息 25 秒后，服务端主动断开；前端 `useWebSocket` Hook 自动重连并回退到 30 秒 HTTP 轮询（`GET /api/v1/matches/live`）。
- Redis Pub/Sub 作为消息总线：爬虫 upsert 后 `PUBLISH data:update {payload}`，WebSocket 消费者广播给前端连接。

## 12. 风险评估与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 懂球帝反爬升级 | 数据采集中断 | 中 | 多数据源冗余；API-Football / FBref 作为备选 |
| API-Football 配额耗尽 | 实时比分中断 | 中 | 降级到懂球帝实时 → 再降级为 Redis 缓存轮询 |
| WebSocket 连接不稳定 | 实时推送中断 | 中 | 前端自动重连 + 30s HTTP 降级轮询；Redis Pub/Sub 解耦 |
| Redis 服务不可用 | 实时缓存失效 | 低 | 降级为直接查 MySQL；Redis 持久化 + 监控告警 |
| Hadoop 环境不稳定 | 数据存储失败 | 低 | 定期备份 HDFS 元数据；MySQL 作为备份存储 |
| 多源数据冲突 | 数据融合困难 | 中 | data_hash 去重 + priority 优先级 + 人工校准关键实体 |
| 世界杯数据不足 | 分析样本不够 | 中 | 补充英超历史数据；Football-Data CSV 补充 |
| 前端开发耗时 | 延误交付 | 高 | 优先核心图表；简化非核心页面 |
| 服务器资源不足 | 性能问题 | 低 | 使用 Parquet 格式优化查询；限制并发爬取 |
